La ambición de este proyecto es transformar \textit{OBS Studio} en una plataforma avanzada de realidad aumentada capaz de integrar de forma nativa elementos tridimensionales y datos dinámicos en transmisiones en vivo. El núcleo del trabajo ha consistido en el diseño y desarrollo de un plugin especializado que extiende las capacidades gráficas del software, permitiendo la superposición de contenido virtual alineado con el mundo físico. A diferencia de otras soluciones de composición externa, este plugin se inserta como una extensión intrínseca en el pipeline de procesamiento de vídeo de \textit{OBS Studio}, garantizando una ejecución de baja latencia y alta eficiencia mediante el uso directo de la API de plugins de OBS y el lenguaje C.

El desarrollo técnico se asienta sobre tres pilares fundamentales. En primer lugar, la implementación de un motor de renderizado basado en \textbf{OpenGL}, capaz de gestionar mallas 3D complejas y materiales en tiempo real. En segundo lugar, la integración de un sistema de detección mediante la biblioteca \textbf{OpenCV}, que utiliza marcadores \textbf{ArUco} y algoritmos de estimación de pose (PnP) para anclar los objetos virtuales a coordenadas reales, compensando incluso la distorsión óptica de la lente mediante una calibración previa. Por último, el plugin incorpora una capa de conectividad asíncrona que, mediante el uso de la biblioteca \textbf{libcurl}, sincroniza el contenido visual con la API de \textbf{DOMjudge}. Esto permite que el sistema no solo renderice gráficos estáticos, sino que proyecte información competitiva en vivo,  clasificaciones, nombres de equipos y estadísticas de resolución de problemas, directamente sobre el entorno de realidad aumentada.

Para facilitar su uso en entornos de producción profesional, el plugin incluye una interfaz de usuario integrada en OBS que permite una configuración exhaustiva. Los usuarios pueden realizar ajustes finos de posición, rotación y escala, seleccionar diferentes diccionarios de marcadores y gestionar la conexión con servidores remotos de concursos de programación DOMJuez de forma intuitiva.


\section{Sistema de Construcción y Ciclo de Ejecución}

El desarrollo del plugin requiere una infraestructura que gestione dependencias multiplataforma y garantice un rendimiento estable en tiempo real dentro de OBS Studio.

\subsection{Infraestructura de Compilación y Dependencias}

Se utiliza \textbf{CMake} como sistema de construcción debido a su capacidad para generar archivos de proyecto nativos en Windows, macOS y Linux. La configuración se apoya en los siguientes elementos clave:

\begin{itemize}
    \item \textbf{CMakePresets:} Define perfiles de compilación estándar (\textit{Debug, Release}), unificando los parámetros de compilación para todos los desarrolladores.
    \item \textbf{Buildspec:} Archivo de metadatos que especifica la versión del plugin, la versión mínima de la API de OBS y la identificación del módulo.
    \item \textbf{Gestión de dependencias:} El proyecto integra tres bibliotecas fundamentales:
    \begin{itemize}
        \item \textbf{OpenCV:} Para el procesamiento de imágenes y la lógica de detección ArUco.
        \item \textbf{ASSIMP:} Para la importación y normalización de modelos 3D (OBJ, FBX).
        \item \textbf{libcurl:} Para la comunicación mediante WebSockets y peticiones HTTP.
    \end{itemize}
\end{itemize}

\subsection{Ciclo de Vida del Plugin}

El ciclo de ejecución se rige por el modelo de eventos de OBS Studio, dividiéndose en tres fases principales:

\subsubsection{Inicialización}
En esta fase se reserva la memoria para la estructura  y se preparan los recursos gráficos:
\begin{enumerate}
    \item \textbf{Carga de Shaders:} Compilación de los programas de GPU necesarios para el renderizado.
    \item \textbf{Inicialización AR:} Carga de los parámetros de calibración de la cámara y configuración del diccionario ArUco.
    \item \textbf{Recursos 3D:} Carga inicial de mallas y texturas en la memoria de vídeo (VRAM).
\end{enumerate}

\subsubsection{Actualización de Lógica }
Este \textit{callback} se ejecuta en cada cuadro (\textit{frame}) para actualizar el estado interno sin bloquear el renderizado:
\begin{itemize}
    \item \textbf{Detección:} Captura del frame de vídeo, conversión a espacio de color BGR y búsqueda de marcadores ArUco.
    \item \textbf{Sincronización:} Procesamiento asíncrono de los datos recibidos vía Web.
    \item \textbf{Cálculo de Pose:} Estimación de la posición y rotación de los objetos 3D basada en los marcadores detectados.
\end{itemize}

\subsubsection{Renderizado Gráfico }
Es la etapa donde se genera la salida visual final. El flujo de trabajo sigue este orden:
\begin{enumerate}
    \item Configuración del \textit{Z-buffer} para la gestión de profundidad.
    \item Aplicación de matrices de transformación (Modelo-Vista-Proyección).
    \item Dibujo de la geometría 3D sobre el flujo de vídeo original de OBS.
    \item Superposición (\textit{Overlay}) de elementos 2D como el marcador de puntuación o el reloj.
\end{enumerate}

\subsection{Configuración Dinámica }

El plugin permite la modificación de parámetros en tiempo de ejecución sin necesidad de reiniciar la fuente de vídeo. Cuando el usuario modifica una propiedad en la interfaz de OBS, el sistema activa una recarga selectiva:
\begin{itemize}
    \item \textbf{Recarga de modelos:} Si cambia la ruta del archivo, se libera la memoria anterior y se carga la nueva geometría mediante ASSIMP.
    \item \textbf{Ajuste de Offsets:} Los cambios en la posición o escala manual se aplican inmediatamente a las matrices de transformación del siguiente frame.
    \item \textbf{Calibración:} Permite ajustar los umbrales de detección AR para adaptarse a diferentes condiciones de iluminación en directo.
\end{itemize}
\section{Carga y Gestión de Modelos 3D}

La visualización de elementos virtuales requiere un sistema capaz de procesar geometría compleja en tiempo real, manteniendo la coherencia espacial con el entorno capturado.

\subsection{Módulo \texttt{SJ\_3DModel}: Gestión de Geometría Poligonal}

El módulo \texttt{SJ\_3DModel} centraliza la lógica de importación y dibujo de objetos. Se ha optado por el formato \textbf{Wavefront OBJ} debido a su amplia compatibilidad y sencillez estructural.

\subsubsection{Estructura de datos de Malla (\textit{Mesh})}
Cada modelo se descompone en una o varias mallas, definidas por:
\begin{itemize}
    \item \textbf{Vertex Buffer:} Almacena las coordenadas $(x, y, z)$, vectores normales para iluminación y coordenadas UV para el mapeo de texturas.
    \item \textbf{Index Buffer:} Define la conectividad de los vértices para formar triángulos, optimizando el uso de memoria GPU.
    \item \textbf{Materiales:} Referencias a texturas externas y propiedades de reflexión de la superficie.
\end{itemize}

\subsection{Transformaciones Espaciales y Control de Usuario}

Para posicionar los modelos, el sistema utiliza **matrices de transformación homogéneas 4x4**. Estas permiten encadenar traslación, rotación y escala en una sola operación matricial:
\[ M = T \cdot R \cdot S \]

\begin{itemize}
    \item \textbf{Rotación (Pitch, Yaw, Roll):} El usuario puede ajustar la orientación manual mediante ángulos de Euler, que internamente se combinan con la matriz de pose calculada por el detector ArUco.
    \item \textbf{Cálculo de Pivotes:} El sistema calcula automáticamente el \textit{Bounding Box} del modelo para situar el centro de rotación en la base o centro geométrico del objeto, evitando rotaciones excéntricas.
    \item \textbf{Escalado Dinámico:} Se implementa un factor de escala global para ajustar el tamaño del modelo virtual al tamaño físico del marcador de referencia.
\end{itemize}



\subsection{Sistema de Texto 3D y Overlays}

El plugin incorpora un motor de renderizado de texto para mostrar información contextual (como el cronómetro o puntuaciones) directamente en la escena.

\subsubsection{Texto Proyectado en el Espacio 3D}
A diferencia de los textos planos, el sistema genera mallas de caracteres que heredan la matriz de transformación del modelo principal. Esto garantiza que el texto:
\begin{enumerate}
    \item Mantenga la perspectiva correcta según el ángulo de la cámara.
    \item Sufra oclusión si un objeto 3D se interpone.
    \item Se desplace y rote en perfecta sincronía con el marcador AR.
\end{enumerate}

\subsubsection{Overlays de Información (HUD)}
Se implementa una capa de visualización 2D (\textit{Heads-Up Display}) para mostrar datos de diagnóstico:
\begin{itemize}
    \item \textbf{Estado de detección:} Indicador visual de si el marcador ArUco está en el campo de visión.
    \item \textbf{Datos de Calibración:} Visualización de los parámetros activos y errores de reproyección.
    \item \textbf{Offsets:} Panel de control para el ajuste fino de la posición visual (X, Y, Z) sobre el vídeo.
\end{itemize}

\section{Detección ArUco y Calibración de Cámara}

La fidelidad de la Realidad Aumentada depende de la capacidad del sistema para traducir coordenadas de píxeles (2D) a una posición en el espacio real (3D). Este proceso se apoya en el uso de marcadores ArUco y una calibración precisa de la óptica.

\subsection{Módulo Detector Aruco: Integración con OpenCV}

Para gestionar la detección, se ha desarrollado el módulo detector Aruco, que actúa como una capa de abstracción sobre la biblioteca \textbf{OpenCV}. Este módulo encapsula la complejidad de la API de OpenCV y expone funciones simplificadas al plugin:

\begin{itemize}
    \item  Analiza el fotograma de OBS en busca de patrones binarios.
    \item  Calcula la traslación y rotación del marcador respecto a la lente.
    \item  Permite seleccionar el diccionario de códigos de detección Aruco al usuario, bajo la interfaz de preferencias de OBS.
\end{itemize}

\subsection{Calibración de la Cámara y Parámetros Intrínsecos}

Cada cámara de vídeo posee imperfecciones ópticas (distorsión radial y tangencial) que curvan la imagen. Para corregirlas, se aplica un modelo matemático de calibración:

\begin{itemize}
    \item \textbf{Matriz Intrínseca ($K$):} Define la distancia focal y el punto principal del sensor. Es vital para que el tamaño del objeto 3D sea coherente con la distancia real.
    \item \textbf{Coeficientes de Distorsión:} Permiten "enderezar" la imagen antes de procesarla, evitando que el modelo 3D "baile" o se deforme en los bordes de la pantalla.
\end{itemize}

El sistema carga estos parámetros desde un archivo generado previamente mediante un proceso de calibración con tablero de ajedrez.

\subsection{Estimación de Pose: El Problema PnP}

Una vez localizado el marcador, el sistema debe resolver el problema de \textbf{Perspectiva-n-Puntos (PnP)}. Este algoritmo compara las 4 esquinas detectadas en la imagen con las coordenadas reales del marcador (que sabemos que es un cuadrado perfecto).

\begin{itemize}
    \item \textbf{Vector de Traslación ($t$):} Indica a qué distancia y en qué dirección está el marcador (X, Y, Z).
    \item \textbf{Vector de Rotación ($R$):} Determina la orientación del marcador. Para que sea legible por el usuario, estos valores se convierten de formato \textit{Rodrigues} a ángulos de \textbf{Euler} (Pitch, Yaw, Roll).
\end{itemize}

\subsection{Robustez y Seguimiento Temporal}

Dada la naturaleza del \textit{streaming} en directo, la detección puede fallar por falta de luz o desenfoque. Se han implementado las siguientes estrategias de mejora:
\begin{enumerate}
    \item \textbf{Filtro de confianza:} Solo se aceptan marcadores con una probabilidad de detección superior a un umbral configurable.
    \item \textbf{Suavizado de Pose:} Se utiliza la información del frame anterior para evitar saltos bruscos en el modelo 3D si la detección oscila levemente entre fotogramas.
\end{enumerate}

\section{Propiedades de Modo AR y Configuración}

La flexibilidad del plugin en entornos de producción en directo depende de un panel de propiedades configurable que permite adaptar la detección y el renderizado a las condiciones físicas del set de grabación.

\subsection{Configuración del Detector: Diccionario e Identificador}

El comportamiento del algoritmo de detección se define mediante dos parámetros críticos que determinan qué debe buscar el sistema en el flujo de vídeo:

\begin{itemize}
    \item \textbf{Selección de Diccionario ArUco:} Permite elegir la complejidad de los marcadores (desde 4x4 hasta 7x7).
    \item \textbf{Identificador de Marcador (ID):} Esta propiedad actúa como un filtro de seguridad. El plugin solo renderizará el contenido 3D cuando detecte el ID específico configurado por el usuario, ignorando otros marcadores que puedan aparecer accidentalmente en la escena.
\end{itemize}

\subsection{Ajustes Finos: Offsets de Posición y Rotación}

Incluso con una calibración perfecta, pueden existir desvíos visuales debido a la inclinación del marcador o errores de precisión del sensor. Para solucionarlo, se han implementado controles de \textit{offset}:

\begin{itemize}
    \item \textbf{Posicionamiento Manual (X, Y, Z):} Permite desplazar el modelo 3D en milímetros respecto al centro del marcador. Es vital para centrar el objeto si el diseño del modelo no coincide exactamente con el centro del patrón ArUco.
    \item \textbf{Rotación Manual (Pitch, Yaw, Roll):} Permite corregir la orientación del modelo. Esto es especialmente útil si el marcador está pegado en una superficie que no es perfectamente horizontal o vertical.
\end{itemize}

\subsection{Modos de Visualización Especiales}

Además del modo AR estándar, el plugin incluye propiedades para gestionar contenidos específicos de concursos de programación:
\begin{itemize}
    \item \textbf{Selección de Malla de Reloj:} Permite elegir qué objeto 3D actuará como cronómetro de la competición.
    \item \textbf{Control de Visibilidad:} El usuario puede activar o desactivar capas de información (\textit{overlays}) como el estado de la conexión web,los datos del detector, modelos, información... Sin interrumpir la emisión.
\end{itemize}


\section{Sincronización Web y Conectividad Remota}

Para que la realidad aumentada sea útil en un concurso de programación, el contenido virtual debe reflejar el estado real de la competición. El plugin incorpora un sistema de sincronización asíncrona que conecta OBS con las APIs de gestión de torneos.

\subsection{Gestión de Conexión y Máquina de Estados}

La comunicación con el servidor se gestiona mediante una **máquina de estados** que garantiza la estabilidad del plugin incluso si la red falla. Esto evita que OBS se congele si el servidor remoto no responde:

\begin{itemize}
    \item \textbf{Activación Binaria:} El usuario puede apagar la sincronización en cualquier momento para trabajar en modo local (\textit{offline}), usando datos en caché.
    \item \textbf{Estados de Conexión:} El sistema transita entre \texttt{DISCONNECTED}, \texttt{CONNECTING}, \texttt{CONNECTED} y \texttt{FAILED}. El estado actual se muestra en el panel de OBS para que el operador sepa si los datos están actualizados.
    \item \textbf{Reintento Automático (\textit{Backoff}):} Si la conexión falla, el plugin no satura la red; espera intervalos crecientes antes de intentar reconectar.
\end{itemize}

\subsection{Configuración de la API y Seguridad}

El acceso a los datos de la competición requiere una configuración precisa del \textit{endpoint} remoto:

\begin{itemize}
    \item \textbf{Autenticación Segura:} Se implementa soporte para \textit{HTTP Basic Auth} y \textit{Tokens}. Las credenciales se almacenan de forma segura aprovechando la API de persistencia de OBS Studio, evitando el uso de texto plano.
    \item \textbf{Identificación del Torneo:} Parámetro clave que filtra los datos del servidor para obtener solo la información del evento actual (ej. un ID de Codeforces).
    \item \textbf{URL Base:} Permite cambiar entre servidores de producción y entornos de prueba sin modificar el código fuente.
\end{itemize}



\subsection{Optimización del Tráfico y Latencia}

Dado que el plugin se ejecuta durante un \textit{streaming} (que ya consume mucho ancho de banda), la eficiencia de red es crítica:

\begin{itemize}
    \item \textbf{Intervalo de Sincronización:} El usuario puede configurar cada cuántos segundos se consultan los datos (ej. cada 30 segundos). Esto permite balancear la "frescura" de la información con el consumo de datos.
    \item \textbf{Gestión de Carga:} El plugin utiliza solicitudes asíncronas para que el hilo de renderizado de vídeo nunca tenga que esperar a la respuesta del servidor (\textit{non-blocking I/O}).
\end{itemize}

\subsection{Presentación Visual: El \textit{Scoreboard} 3D}

Una vez obtenidos los datos (puntuaciones, nombres, posiciones), el sistema los proyecta en el espacio virtual:

\begin{itemize}
    \item \textbf{Offset del Scoreboard:} Permite mover la tabla de puntuaciones en la pantalla de OBS para que no tape elementos críticos del vídeo. Además de utilizar el AR para posicionar el Scoreboard donde interese
    \item \textbf{Modos de visualización:} El plugin ofrece dos vistas diferenciadas que el operador selecciona según el momento de la retransmisión. En el modo Scoreboard se presenta la clasificación general del concurso, con posición, nombre, problemas resueltos y tiempo de penalización de cada equipo. En el modo Team Info el contenido se focaliza en el equipo que la cámara está captando en ese instante, mostrando únicamente sus datos mientras el marcador sea visible.
\end{itemize}

\subsection{Arquitectura de Red Asíncrona (\texttt{libcurl})}

Para evitar que la interfaz de OBS Studio se bloquee mientras se esperan datos del servidor, se ha implementado un módulo de red basado en la biblioteca \textbf{libcurl}.

\begin{itemize}
    \item \textbf{Comunicación No Bloqueante:} Las peticiones HTTP se realizan en un hilo secundario dedicado que corre en paralelo al renderizado. El hilo principal de OBS nunca espera la respuesta del servidor; simplemente lee el último resultado disponible cuando lo necesita.
    \item \textbf{Gestión de Errores:} El sistema implementa políticas de \textit{timeout} y reintentos exponenciales. Si el servidor DOMjudge cae, el plugin sigue funcionando con los últimos datos guardados en caché.
\end{itemize}



\subsection{Consumo de Endpoints y Procesamiento JSON}

El plugin consulta periódicamente tres puntos clave de la API REST de DOMjudge para reconstruir el estado de la competición:

\begin{enumerate}
    \item \textbf{contests}: Obtiene metadatos como el nombre del torneo y el tiempo restante (vital para sincronizar el cronómetro 3D).
    \item \textbf{teams}: Descarga la lista de participantes, sus nombres reales y sus afiliaciones universitarias.
    \item \textbf{scoreboard}: Es el recurso más crítico. Proporciona el ranking actual, los problemas resueltos y el tiempo de penalización de cada equipo.
\end{enumerate}

El módulo \texttt{json\_utils} se encarga de deserializar estas respuestas (formato JSON) y convertirlas en estructuras de datos de lenguaje C, listas para ser enviadas al motor de renderizado.

\subsection{Sincronización AR: Mapeo de Marcadores a Equipos}

La funcionalidad más innovadora del proyecto es el vínculo entre el mundo físico y los datos digitales. Esto se logra mediante una **tabla de mapeo**:

\begin{itemize}
    \item \textbf{Identificación Unívoca:} El operador asigna en las propiedades del plugin qué \textit{ID de Marcador ArUco} corresponde a qué \textit{ID de Equipo} en DOMjudge.
    \item \textbf{Visualización Contextual:} Cuando la cámara detecta el marcador de un equipo, el plugin recupera automáticamente sus datos y los muestra como un \textit{overlay} de texto sobre el vídeo, indicando su posición en el ranking, su nombre y el número de problemas resueltos.
\end{itemize}



\subsection{Presentación de Datos y Modos de Visualización}

El sistema alterna automáticamente entre dos modos de visualización para enriquecer la narrativa de la retransmisión:

\begin{itemize}
    \item \textbf{Modo General (Scoreboard):} Si no hay marcadores en pantalla, se muestra una tabla clásica de posiciones con el Top 10 de la competición.
    \item \textbf{Modo Foco (Equipo Detectado):} Al detectar un marcador, el sistema hace un "zoom" visual a los datos de ese equipo, mostrando detalles de sus envíos (envíos aceptados, intentos fallidos y veredictos en tiempo real).
\end{itemize}

\section{Renderizado de Overlays 2D Anclados al Marcador}
\label{sec:overlays2d}

Los modos Scoreboard y Team Info comparten una misma arquitectura de renderizado cuyo propósito es diferente al del modelo 3D: en lugar de volumen y geometría, lo que se superpone al vídeo es un bloque de texto con información de la competición, diseñado para parecer físicamente anclado al marcador. El texto se mueve y rota con él cuando la cámara se desplaza, sin temblores perceptibles y sobre un fondo visualmente limpio. Lo que distingue a ambos modos entre sí es la fuente de datos y el criterio de detección del marcador, no el mecanismo de renderizado, que es común.

\subsection{El Sistema de Coordenadas: Por Qué Todo Funciona con el Origen Arriba}
\label{subsec:ydown}

Al combinar visión por computador con un motor gráfico surge casi inevitablemente un problema de convenios: OpenCV sitúa el origen de coordenadas en la esquina superior izquierda de la imagen con el eje vertical creciendo hacia abajo, mientras que la tradición de OpenGL lo coloca abajo a la izquierda con el eje vertical hacia arriba. Si no se resuelve, cada posición que devuelve el detector aparece invertida verticalmente en pantalla.

La solución adoptada fue configurar la proyección de OBS explícitamente con el convenio de OpenCV —origen arriba a la izquierda, eje $Y$ hacia abajo—, de modo que las coordenadas del marcador se pueden usar directamente sin ninguna transformación adicional. Esto elimina una fuente frecuente de errores y hace el código mucho más fácil de razonar. Como efecto secundario, fue necesario configurar un rango de profundidad muy amplio en la proyección para evitar que el texto se recortara cuando el marcador se inclina y el panel de texto sale del plano frontal, un problema que no era evidente hasta que se probaron ángulos de marcador extremos en directo.

\subsection{Qué se Quiere Lograr Visualmente y Cómo se Consigue}
\label{subsec:objetivo_overlay}

El efecto buscado es que el bloque de texto parezca estar encima del marcador como una pantalla flotante sobre la mesa. Si el marcador está inclinado hacia la cámara, el texto se inclina con él. Si el marcador gira en el plano de la mesa, el texto gira también. Y si el operador mueve la cámara, el texto permanece centrado sobre el marcador en todo momento.

Para lograr esto, el sistema calcula en cada frame el centro geométrico del marcador en coordenadas de pantalla, promediando las posiciones de sus cuatro esquinas detectadas. Ese punto se convierte en el origen de toda la cadena de transformaciones que posicionará el texto. A partir de ahí se aplica la orientación real del marcador en el espacio —extraída por el algoritmo de estimación de pose— y un ajuste final que centra el bloque de texto sobre ese punto de anclaje. El usuario puede afinar el resultado con controles de offset de posición y rotación que actúan siempre sobre los ejes de la pantalla, independientemente de cómo esté orientado el marcador, lo que hace que su comportamiento sea intuitivo incluso para alguien que no conozca los detalles de la implementación.

\subsection{El Módulo de Detección en los Modos de Texto}
\label{subsec:deteccion_overlay}

La detección se ejecuta en el hilo de vídeo de OBS, separado del hilo de renderizado, de modo que el análisis de imagen no bloquea en ningún momento la salida visual. Cada frame que llega de la fuente de vídeo se convierte al formato de color interno —el módulo de detección soporta todos los formatos habituales que OBS puede entregar, desde YUV420 hasta BGRA— y se pasa al detector en escala de grises.

El detector localiza los marcadores presentes en la imagen y, para el elegido, calcula su posición y orientación tridimensional respecto a la cámara mediante el problema de Perspectiva-$n$-Puntos. La idea es sencilla: si se sabe que el marcador es un cuadrado de tamaño conocido y se ven sus cuatro esquinas en la imagen, existe una única posición y orientación en el espacio que explicaría exactamente esa proyección. El resultado son un vector de traslación (a qué distancia y en qué dirección está el marcador) y un vector de rotación (cómo está orientado en el espacio).

Las coordenadas en píxeles de las esquinas se escalan a la resolución de salida de OBS de forma proporcional:

\[
  s_x = \frac{W_{\text{base}}}{W_{\text{frame}}}, \qquad s_y = \frac{H_{\text{base}}}{H_{\text{frame}}}
\]

donde $W_{\text{base}} \times H_{\text{base}}$ es la resolución base de la escena OBS y $W_{\text{frame}} \times H_{\text{frame}}$ la del frame capturado. Este escalado garantiza que las posiciones del marcador sean siempre coherentes con el espacio en que se dibuja el overlay, independientemente de la resolución real de la cámara.

\subsubsection{De la Rotación Detectada a la Orientación del Overlay}
\label{subsubsec:rodrigues}

La orientación que devuelve el algoritmo viene en un formato llamado representación de Rodrigues: un único vector donde la dirección indica el eje alrededor del que gira el marcador y la longitud indica cuánto gira. Es una forma compacta y sin ambigüedades de codificar cualquier rotación en el espacio, pero para orientar el overlay se necesita traducirla a algo que OBS pueda aplicar directamente.

El módulo de detección convierte esa representación a una matriz de rotación $3\times3$ que describe cómo están orientados los tres ejes del marcador respecto a la cámara. De esa matriz se extraen los ángulos de inclinación hacia delante y hacia los lados (pitch y yaw) y el giro sobre el propio eje (roll), que son los que finalmente guían cómo se orienta el panel de texto en el espacio 3D de OBS. Cuando el marcador se inclina extremadamente, existe un caso matemático conocido llamado \textit{gimbal lock} donde dos de los tres ángulos se vuelven dependientes y no se puede distinguirlos por separado; el sistema detecta esta situación y aplica un cálculo alternativo para evitar comportamientos extraños en el overlay.

Adicionalmente, el módulo calcula en qué dirección apunta la cara del marcador proyectada sobre la imagen de la cámara. Esto sirve para que el texto siempre resulte legible: aunque el marcador esté ligeramente girado en el plano de la mesa, el sistema ajusta la orientación del overlay para que el texto no quede volcado o boca abajo.

\subsection{Selección del Marcador en el Modo Team Info}
\label{subsec:seleccion_marcador}

En el modo Scoreboard el sistema busca un único marcador con el ID configurado. En el modo Team Info cualquier marcador registrado en el JSON de mapeo es válido. Si varios marcadores válidos aparecen en cámara al mismo tiempo, el sistema usa el primero que encuentra en el orden en que el detector los devuelve, que coincide con el orden natural de aparición en la imagen. Este comportamiento es simple y predecible: en un escenario habitual de retransmisión la cámara apunta a un equipo concreto, por lo que la situación de ambigüedad raramente se presenta. Si en algún momento se necesita mostrar información de varios equipos simultáneamente, la arquitectura de plugins de OBS lo resuelve añadiendo una instancia del filtro por cada equipo, cada una configurada para su marcador correspondiente, trabajando en paralelo sin interferirse.

\subsection{La Composición Visual del Scoreboard}
\label{subsec:multicolumna}

El modo Scoreboard organiza la información en cuatro columnas: posición, nombre del equipo, problemas resueltos y tiempo de penalización. Cada columna es una fuente de texto de OBS independiente, lo que permite controlar el estilo visual de cada una por separado —tamaño de fuente, color, negrita— y que sus anchos se adapten automáticamente al contenido real en cada momento, sin riesgo de que un nombre largo desborde o quede truncado artificialmente.

El ancho total del bloque se calcula sumando los anchos reales de las cuatro columnas más tres separadores fijos:

\[
  T_w = W_{\text{pos}} + W_{\text{nombre}} + W_{\text{res}} + W_{\text{tiempo}} + 3\,\delta, \qquad \delta = 18\;\text{px}
\]

Las cuatro columnas se dibujan en secuencia de izquierda a derecha, avanzando un cursor horizontal que suma el ancho de cada columna recién renderizada antes de dibujar la siguiente. El resultado visual es una tabla compacta que, al estar todas sus columnas bajo la misma transformación AR, se mueve y rota como un bloque único cuando está anclada al marcador.

\subsection{La Cadena de Transformaciones para Anclar el Texto al Marcador}
\label{subsec:pipeline_matrices}

Anclar el bloque de texto al marcador con la orientación correcta requiere encadenar varias transformaciones en un orden muy preciso. La API gráfica de OBS gestiona estas transformaciones mediante un stack acumulativo: cada nueva operación se combina con todas las anteriores y el vértice final recibe el resultado completo.

El objetivo visual, ya descrito en la sección~\ref{subsec:objetivo_overlay}, exige que el texto quede centrado sobre el marcador, erguido perpendicularmente a su superficie y orientado siguiendo la rotación real del marcador en el espacio. Para conseguirlo, la cadena de transformaciones parte del punto más global al más local:

En primer lugar, se traslada el origen de dibujo al centro del marcador en pantalla, incorporando ya el offset de posición que el usuario haya configurado. Sobre ese punto se aplican los offsets de rotación manuales del usuario; gracias al orden en que actúan dentro del stack, estos siempre giran el panel sobre los ejes de la pantalla (inclinación, giro lateral, giro en profundidad) con independencia de cómo esté orientado el marcador físico, lo que hace que su efecto sea predecible e intuitivo. A continuación se corrige la diferencia de convenios entre el sistema de coordenadas de OpenCV y el de OBS, que difieren en la dirección del eje vertical, mediante una rotación de $180°$ sobre el eje horizontal. Después se aplica la rotación real del marcador detectada por el algoritmo PnP, que es la que hace que el texto se incline y gire cuando lo hace el marcador físico. Una vez orientado así el panel, se rota $90°$ adicionales para que quede erguido sobre el marcador, como un cartel sobre una mesa, en lugar de estar tumbado sobre él. Por último, el bloque se desplaza la mitad de su ancho hacia la izquierda y su altura completa hacia arriba, de modo que quede centrado horizontalmente sobre el marcador y crezca hacia arriba desde el punto de anclaje.

La transformación total que recibe cada vértice del overlay, expresada como composición de todos estos pasos, es:

\[
  M = T_{\text{pos}} \cdot R_{\text{offsets}} \cdot R_{\text{flip}} \cdot R_{\text{ArUco}} \cdot R_{\text{elevación}} \cdot T_{\text{centrado}}
\]

\subsection{Suavizado del Overlay para Evitar el Temblor}
\label{subsec:suavizado}

Aunque la detección ArUco es robusta, el ruido de la imagen de cámara —granulado del sensor, compresión de vídeo, microvibraciones— provoca que la posición y el ángulo calculados del marcador oscilen ligeramente de frame a frame. Sin ninguna corrección, el texto parece temblar sobre el vídeo, lo que resulta poco profesional en una retransmisión.

Para mitigarlo, el sistema aplica un filtro de suavizado exponencial sobre la posición y el ángulo del overlay. La idea es sencilla: en lugar de saltar directamente al valor nuevo detectado en cada frame, se mezcla ese valor nuevo con el del frame anterior según un factor configurable $\alpha \in (0,\,1]$:

\[
  P_t = (1 - \alpha)\,P_{t-1} + \alpha\,P_{\text{detectado}}
\]

Con $\alpha$ cercano a $1$ el overlay sigue el marcador con mucha fidelidad pero puede mostrar algo de temblor residual. Con $\alpha$ más pequeño el movimiento es más suave pero tiene algo más de inercia, por lo que en cámara rápida puede quedarse ligeramente retrasado. El operador ajusta este parámetro según las condiciones de grabación.

El ángulo requiere un cuidado adicional: si el marcador pasa de estar a $179°$ a $-179°$, una interpolación directa haría girar el overlay $358°$ en sentido contrario en lugar de $2°$. El sistema detecta este caso y siempre interpola por el camino angular más corto entre los dos valores. El suavizado se reinicia automáticamente cuando el marcador detectado cambia de ID, para no arrastrar la posición del marcador anterior a la animación del nuevo.

\subsection{Fondo Visual del Overlay}
\label{subsec:rounded}

Para mejorar la legibilidad del texto sobre vídeo en movimiento, el sistema puede dibujar un fondo semitransparente detrás del bloque de texto, con esquinas redondeadas y una sombra opcional. Las esquinas redondeadas se generan como arcos de círculo, aproximando cada esquina con varios segmentos cortos cuya longitud depende del radio configurado. Esta geometría se calcula una vez y se sube a la memoria de la tarjeta gráfica, donde queda almacenada para los frames siguientes sin necesidad de recalcularla. Solo se regenera si el tamaño del bloque de texto cambia de forma apreciable, lo que en la práctica ocurre raramente durante una retransmisión estable.

La sombra se consigue dibujando el mismo fondo varias veces con un pequeño desplazamiento incremental y con opacidad decreciente en cada pasada. No requiere shaders adicionales y produce un efecto de profundidad visual convincente con un coste computacional muy reducido.

\subsection{El JSON de Mapeo en el Modo Team Info}
\label{subsec:teaminfo}

El modo Team Info usa un fichero JSON que el operador prepara antes de la retransmisión para asociar cada marcador físico impreso en las mesas del concurso con el identificador de equipo en DOMjudge. Su formato es una lista de pares:

\begin{verbatim}
[
  { "marker_id": 1, "team_id": "42" },
  { "marker_id": 2, "team_id": "12" },
  { "marker_id": 5, "team_id": "7"  }
]
\end{verbatim}

Al cargar el fichero, el módulo de sincronización construye en memoria la tabla de mapeos y extrae también la lista compacta de todos los IDs de marcador válidos. Esta lista es la que se pasa al módulo de detección para que, en cada frame, filtre los marcadores detectados y descarte los que no corresponden a ningún equipo registrado. Una sola pasada de detección es suficiente para encontrar todos los candidatos válidos y seleccionar el más prominente.

Una vez identificado el marcador, el sistema cruza su ID con la tabla de mapeo para obtener el identificador del equipo y busca sus datos en la caché local, que se actualiza periódicamente con la información descargada de la API de DOMjudge. Si el equipo está en caché, su información —nombre, posición en el ranking, problemas resueltos y tiempo de penalización— se vuelca en el bloque de texto del overlay y se renderiza con el mismo mecanismo de posicionamiento descrito en las secciones anteriores.

\section{Liberación de Recursos y Finalización}

Un aspecto crítico en el desarrollo de plugins para OBS Studio es la gestión del ciclo de vida de los recursos. Debido a que el plugin puede activarse y desactivarse dinámicamente, es imperativo garantizar una limpieza absoluta para evitar fugas de memoria (\textit{memory leaks}) que degraden el rendimiento del sistema anfitrión.

\subsection{Seguridad y Persistencia de Estado}

Antes del cierre definitivo, el plugin realiza dos tareas finales de integridad:
\begin{enumerate}
    \item \textbf{Limpieza de Credenciales:} Por seguridad, las contraseñas de la API se sobrescriben en memoria antes de ser liberadas, evitando que queden trazas en el \textit{heap} del proceso.
    \item \textbf{Guardado de Configuración:} Los \textit{offsets} y ajustes realizados por el usuario durante la sesión se persisten en el archivo de configuración de la escena de OBS, permitiendo que la próxima vez que se cargue el plugin, el alineamiento sea idéntico.
\end{enumerate}

\subsection{Gestión de Memoria GPU (Mallas y Texturas)}

Los recursos gráficos residen en la memoria de la tarjeta de vídeo y su liberación debe estar sincronizada con el hilo de renderizado de OBS:

\begin{itemize}
    \item \textbf{Destrucción de Buffers:} Al desactivar el plugin, se invocan de forma recursiva las funciones de destrucción de los \textit{Vertex Buffer Objects} (VBO) y los \textit{Index Buffers}. Se sigue un orden estricto para evitar accesos a memoria inválida.
    \item \textbf{Limpieza de Texturas:} Las imágenes cargadas para los modelos 3D y el reloj se eliminan de la VRAM. Se utiliza un sistema de conteo de referencias para asegurar que una textura compartida solo se borre cuando ningún objeto la necesite.
\end{itemize}

\subsection{Cleanup del Módulo de Detección (OpenCV)}

El detector ArUco mantiene estructuras pesadas que deben cerrarse ordenadamente:
\begin{itemize}
    \item \textbf{Estructuras de OpenCV:} Se liberan los diccionarios de marcadores y las matrices de calibración de cámara para devolver la memoria al sistema.
    \item \textbf{Seguimiento Temporal:} Se reinicializan los filtros de suavizado y las últimas poses conocidas, garantizando que si el plugin se vuelve a activar, comience desde un estado limpio sin heredar posiciones antiguas.
\end{itemize}

\subsection{Cierre de Conexiones de Red y Sincronización}

El módulo de sincronización web gestiona recursos externos que requieren una finalización controlada. Si hay una petición a DOMjudge en curso al cerrar el plugin, el sistema señaliza al hilo secundario que debe detenerse, espera a que termine limpiamente y cierra los sockets TCP abiertos. A continuación se liberan todas las estructuras de datos que contienen el scoreboard, la lista de equipos, los resultados de los problemas y la tabla de mapeo de marcadores, recorriendo los arrays dinámicos de forma recursiva hasta dejar la memoria completamente limpia.
